在二维空间xoy坐标系中绘制三角形并旋转

（第一部分：采用固定管线模式）

作者：刘文庆

本程序我们完成一个二维空间xoy坐标系下绘制三角形的程序设计，我们主要学习基于OpenGL Mathematics 库的使用和二维平面下坐标变换的基本方法，在编程过程中我们分为两个步骤，第一步我们学习通过传统的方法绘制三角形和坐标轴，第二步我们通过VBO绘制三角形，学习现代VBO的概念和使用方法，为我们后续的基于Shader的绘制过程创建VBO/VAO/EBO 打下一定的基础。

OpenGL版本变迁： GL.h、glu.h头文件(opengl32.lib、glu32.lib)只实现了OpenGL1.1版本定义的函数，从1.2开始定义的函数称为OpenGL扩展，标准委员会通过glext.h头文件及说明文档向外公布，具体实现依赖计算机显卡硬件及其驱动程序，需要通过函数指针形式查询本机显卡是否支持，引入函数体后才能在程序中使用，所以出现了多个引入OpenGL扩展的库，比较著名的有： GLEW 、glee 、glad 等。其中本文提到的glee为作者以前使用过，并且根据网上提供的源代码扩展升级到了OpenGL4.6版本。目前glad库使用的比较多，所以作者已经将glad库集成到了WinFormOpenGL.zip和WinFormGLFWApp.zip向导程序中，并且在bool CreateGLContent(HWND hWnd, const float c_GLVersion = 3.2f);函数中自动为我们引出了所有OpenGL扩展函数，我们再程序中直接使用就可以了。

OpenGL 2.0之前基本只能编写传统类型的OpenGL程序，我们称为固定管线渲染OpenGL程序，即OpenGL代码如下： 

1
 glBegin(GL_TRIANGLES);
2
     glVertex2f(verts[0].x, verts[0].y);
3
     glVertex2f(verts[1].x, verts[1].y);
4
     glVertex2f(verts[2].x, verts[2].y);
5
 glEnd();

OpenGL 2.0引入了GLSL概念（对应Shader概念），即通过程序编写运行在显卡GPU中的渲染代码，提高显卡的运行效率和OpenGL程序渲染的灵活性，但是由于OpenGL设计的三维程序渲染流程相对固定，当时GLSL编程并没有被工业界广泛接受；OpenGL 2.0同时引入了glBindBuffer(GL_ARRAY_BUFFER, VBO);函数，提出了顶点缓冲区VBO(vertex buffer object)的概念，通过程序一次性将顶点数据有内存传输到显卡显存中，显著的提高了OpenGL应用程序执行的效率，特别是复杂场景中大数据显卡传输问题。

OpenGL 3.0为了提高VBO的程序管理，引入了顶点缓冲对象VAO(Vertex Array Objects）的概念，即用于保存管理VBO的缓冲区，同时要求VAO对象的绘制代码必须使用GLSL(Shader)才能正确绘制。OpenGL 3.0、3.1、3.2为了兼容传统的固定管线渲染代码，提出了WGL_CONTEXT_COMPATIBILITY_PROFILE_BIT_ARB兼容模式的GLContent程序工作环境，在兼容模式下绘制代码既可以使用传统的glBegin/glEnd绘制图形，也可是使用Shader+VBO/VAO/EBO形式绘制图形。OpenGL3.3以后舍弃的兼容模式，全面使用WGL_CONTEXT_CORE_PROFILE_BIT_ARB核心模式，应用程序只能使用Shader+VBO/VAO/EBO形式渲染图形。

关于GLSL(Shader)我们将单独进行讲解，关于VBO/VAO/EBO相关概念我们会分步骤的结合例子进行讲解，本节后半部分我们重点讲解VBO的构建过程及使用方法。

③ 控制三角形旋转和平移，理解GLM库定义的矩阵的存储循序为列优先循序，理解GLM库计算过程中行优先和列优先的逻辑关系

知识点：glm库存储矩阵元素使用一维数组存储的即：mat[4][4]矩阵存储为T* Data = new T[16]；的一维数组中。

而矩阵元素的存储顺序为列优先的形式：即

mat[0][0]=Data[0]mat[0][1]=Data[4]mat[0][2]=Data[8]mat[0][3]=Data[12]mat[1][0]=Data[1]mat[1][1]=Data[5]mat[1][2]=Data[9]mat[1][3]=Data[13]mat[2][0]=Data[2]mat[2][1]=Data[6]mat[2][2]=Data[10]mat[2][3]=Data[14]mat[3][0]=Data[3]mat[3][1]=Data[7]mat[3][2]=Data[11]mat[3][3]=Data[15]

glm库列优先的存储形式与OpenGL内部使用的计算矩阵存储模式完全一致，之方便了矩阵数据传输到GPU中的传输过程，不需要数据位置转化。

编程过程中很多资料描述GLM库使用的是行向量乘法对应OpenGL内部完全相同的计算方法，而微软的DirectMath对应Direct3D所使用的列向量乘法 。如何理解这个概念，下面我们详细进行描述：

所谓行向量乘法与列向量乘法是一个逻辑上的概念，是指计算过程中矢量v=｛x,y,z,1｝是看作列向量还是行向量的问题，采用行向量乘法的时候v=｛x,y,z,1｝作为列向量参与计算，为矩阵mat中的每一列对应X、Y、Z坐标系中的单位基，即mat的理论形式为：

mat=(XxYxZxMovXXyYyXyMovYXzYzZzMovZ0001)

所以OpenGL内部计算过程应该表示为：

(x′y′z′1)=(xyz1)(XxYxZxMovXXyYyXyMovYXzYzZzMovZ0001)

而微软的DirectMath对应Direct3D所使用的列向量乘法正好相反。
